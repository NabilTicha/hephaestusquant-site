Infrastructure

Kernel

Python-based research and backtesting engine. Built for factor research, not for trading. Open-source when it's ready.

In development
Why

The problem

Platforms like QuantConnect and Zipline make assumptions about strategy structure, data access, and execution that get in the way of research. They're built to deploy strategies, not to study them. If you want to do something like run a cross-sectional momentum sort with regime-conditional rebalancing and custom transaction cost models, you end up fighting the framework.

Kernel is the opposite: research-first, with every component exposed and replaceable. No black boxes, no vendor lock-in.

Design

How it's built

Modular
Data ingestion, factor construction, backtesting, and risk analytics are independent modules. Replace any part without touching the rest.
Reproducible
Every research run is deterministic and versioned. Same inputs + same config = same output. We use hashed pipeline configs to track this.
Transparent
No proprietary layers. All logic is readable source. Assumptions are documented in pipeline YAML, not buried in code.
Python 3.11+
Built on NumPy, pandas, and statsmodels. Arch for GARCH modeling. hmmlearn for regime detection. Nothing exotic.
Modules

Current scope

  • Data Point-in-time data ingestion with survivorship-bias handling. Currently supports CRSP/Compustat. Refinitiv Datastream adapter is in progress.
  • Factors Cross-sectional and time-series factor construction framework. Supports custom sort/portfolio construction. Winsorization, industry-neutralization built in.
  • Backtest Vectorized backtesting with configurable cost models. Event-driven mode planned but not yet implemented.
  • Risk Factor/idiosyncratic/regime risk decomposition.
  • Regime Rolling regime detection via 2-state Gaussian HMM on realized volatility. Markov-switching and threshold models are next.
Status

Where things stand

Kernel is under active internal development. Core modules (data, factors, vectorized backtest) are functional and used in our research. Risk and regime modules are earlier-stage.

We'll open-source it when the API is stable enough that we won't need to break backwards compatibility every two weeks. Not there yet.
